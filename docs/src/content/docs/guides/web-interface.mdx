## Overview

The UseCallManager web interface provides a modern, user-friendly dashboard for managing TVS and CAPF services. This guide walks through every page with detailed screenshots and explanations.

## Dashboard

The main dashboard provides an at-a-glance view of your entire system:

### Statistics Cards

**Top Row - Quick Metrics:**
- **Certificates**: Total number of trusted certificates in TVS
- **Devices**: Total enrolled devices in CAPF
- **Pending Operations**: Count of devices awaiting certificate enrollment/deletion
  - Breakdown: Install, Fetch, Delete operations
- **Active Connections**: Current phone connections to TVS and CAPF services

### Service Health

**Real-time Status Monitoring:**
- **TVS (Trust Verification)**: Shows healthy/offline status with green/red indicator
  - Displays protocol port (2445)
  - Status badge shows "healthy" when service responds
- **CAPF (Certificate Authority)**: Certificate enrollment service status
  - Displays protocol port (3804)
  - Status badge with health indicator

**Indicators:**
- ðŸŸ¢ Green dot + "healthy" badge = Service operational
- ðŸ”´ Red dot + "Offline" badge = Service unreachable

### Quick Actions

**Primary Actions:**
- **Setup Wizard** (highlighted in blue): Launch onboarding flow for initial configuration
- **Upload Certificate**: Add new trusted certificates to TVS
- **Add Device**: Schedule certificate operations for phones

### Error Display

When services are offline, a red-bordered card appears showing:
- Connection error details for each service
- Helpful message: "Make sure the backend services are running and accessible"

## Onboarding Wizard

The setup wizard guides you through initial system configuration in four steps.

### Step 1: Welcome



**Welcome Screen:**
- Introduction to UseCallManager services
- Explains TVS and CAPF functionality
- Overview of certificate-based security
- "Next" button to proceed

**What You'll Learn:**
- What TVS (Trust Verification Service) does
- What CAPF (Certificate Authority Proxy Function) does
- Why certificate security matters for IP phones

### Step 2: Upload CA Certificate



**Certificate Upload Interface:**
- **File Upload Zone**: Drag-and-drop or click to select
  - Accepts PEM-encoded X.509 certificates
  - Shows upload progress
- **Certificate Roles**: Multi-select checkboxes
  - CCM (Call Manager)
  - TFTP (Configuration Server)
  - CAPF (Certificate Authority)
  - Other roles for specialized services
- **Purpose**: Upload your CallManager/server certificates so phones can validate them

**Supported Formats:**
- PEM (.pem, .crt)
- DER (.der, .cer)
- Base64 encoded certificates

### Step 3: Configure Device



**Device Configuration Form:**
- **Device Name**: SEP + MAC address (e.g., SEP001122334455)
  - Auto-formatted to uppercase
  - Validates MAC address format
- **Operation**: Dropdown selection
  - Install: Generate new certificate
  - Fetch: Retrieve existing certificate
  - Delete: Remove certificate
- **Authentication**: Security method
  - No Password: Least secure, testing only
  - By Null String: Empty auth string
  - By Password: Required password (recommended)
- **Password Field**: Appears when "By Password" selected
  - Used by phone during enrollment
  - Minimum 8 characters recommended
- **Key Size**: RSA key strength
  - 512, 1024, 2048, 3072, 4096 bits
  - 2048 recommended for most phones

**Validation:**
- Device name format checked
- Password strength indicator
- Required fields highlighted

### Step 4: Completion



**Setup Complete:**
- âœ… Summary of configuration
- Certificate uploaded confirmation
- Device scheduled for enrollment
- **Next Steps** guidance:
  - Configure phone to point to CAPF
  - Trigger enrollment from phone menu
  - Monitor audit logs for progress
- **Finish** button returns to dashboard

## Certificates Page



**Certificate Management:**

### Page Header
- **Title**: "Certificates" with certificate count badge
- **Upload Certificate** button: Add new certificates

### Certificate List

**Empty State:**
When no certificates are uploaded, shows:
- Icon indicating empty state
- "No certificates found" message
- "Upload Certificate" call-to-action button
- Helpful text: "Upload your first certificate to get started"

**With Certificates (populated state):**
Each certificate card displays:
- **Certificate Hash**: First 16 characters of SHA-256 hash
- **Subject Name**: Certificate DN (Distinguished Name)
- **Issuer**: Who signed the certificate
- **Serial Number**: Unique certificate identifier
- **Roles**: Badges showing assigned roles (CCM, TFTP, etc.)
- **Actions**:
  - View Details: Expand full certificate information
  - Download: Export certificate as PEM
  - Delete: Remove from trusted list

### Upload Dialog

**File Upload:**
- Drag-and-drop zone
- Click to browse files
- Preview selected file

**Role Selection:**
- Checkboxes for each role type
- Multiple roles can be selected
- Tooltips explain each role

**TTL (Time-to-Live):**
- Cache duration in seconds
- Default: 86400 (24 hours)
- Phones cache validation results for this period

## Security Tools



**Advanced Security Utilities:**

### ITL (Initial Trust List) Generator

**Purpose**: Create ITL files for Cisco phone secure boot

**Configuration:**
- **Phone Models**: Select target phone types
  - 7900 series
  - 8800 series
  - 9900 series
- **Certificate List**: Choose which certificates to include
- **Signature**: Sign with CAPF private key
- **Generate**: Create ITL file for TFTP deployment

**Output:**
- Binary ITL file
- Ready for TFTP server
- Phones download on boot

### Config Encryptor

**Purpose**: Encrypt sensitive phone configuration data

**Features:**
- **Input**: Plain XML configuration
- **Encryption**: AES-256-GCM
- **Key Management**: Auto-generated or custom key
- **Output**: Encrypted config blob

**Use Cases:**
- Protect passwords in phone configs
- Secure TFTP transmission
- Compliance requirements

### Certificate Validator

**Purpose**: Verify certificate chains and validity

**Checks:**
- Certificate signature validation
- Chain of trust verification
- Expiration status
- Key usage compliance
- CRL/OCSP checking

**Results:**
- âœ… Valid: Certificate passes all checks

## Administration

### TVS Audit Logs



**Certificate Validation Audit Trail:**

### Log Filters

**Filter Options:**
- **Operation Type**:
  - All Operations
  - Validate: Certificate validation requests
  - Upload: Certificate additions
  - Delete: Certificate removals
- **Success Status**:
  - All
  - Success only
  - Failures only
- **Date Range**: Filter by time period
- **Client IP**: Filter by requesting IP address

### Log Entries

**Each Entry Shows:**
- **Timestamp**: Exact date and time (with timezone)
- **Client IP**: Phone or system that made request
- **Operation**: What action was performed
- **Resource**: Certificate hash or identifier
- **Status**: Success (green) or Failed (red)
- **Details**: Expandable error messages or metadata

**Common Operations:**
- `validate`: Phone checking certificate validity
- `upload`: New certificate added via web UI
- `delete`: Certificate removed from trusted list
- `system`: Service startup/shutdown events

### Export Functionality

**Export Logs:**
- CSV format for spreadsheet analysis
- JSON format for programmatic processing
- Filtered results only
- Full audit trail export

### CAPF Audit Logs



**Device Enrollment Audit Trail:**

### Log Filters

**Filter Options:**
- **Operation Type**:
  - All Operations
  - Enroll: Certificate enrollment attempts
  - Update: Device configuration changes
  - Delete: Device removal
- **Success Status**: All/Success/Failures
- **Date Range**: Time-based filtering
- **Device Name**: Specific device lookup

### Log Entries

**Each Entry Shows:**
- **Timestamp**: When operation occurred
- **Client IP**: Phone IP address
- **Operation**: Type of action
- **Resource**: Device name (SEP...)
- **Status**: Success/Failed with color coding
- **Error Message**: Failure reason (if applicable)
- **Metadata**: Additional context (JSON)

**Common Operations:**
- `enroll`: Phone connected for certificate
- `authenticate`: Password verification attempt
- `install`: Certificate successfully installed
- `delete`: Certificate removed from phone
- `schedule`: Operation scheduled via web UI

**Failure Reasons:**
- Authentication failed: Wrong password
- Connection timeout: Network issue
- No pending operation: Phone not scheduled
- Certificate generation error: Crypto failure

### Troubleshooting

**Use Audit Logs to:**
- Verify phones are contacting CAPF
- Identify authentication failures
- Track enrollment success rates
- Debug network connectivity issues
- Monitor for suspicious activity

## Common Workflows

### Initial Setup

1. **Start Onboarding Wizard** (Dashboard â†’ Setup Wizard)
2. **Upload CA Certificate** (Step 2)
   - Select CallManager certificate
   - Choose roles: CCM, TFTP
3. **Configure First Device** (Step 3)
   - Enter device MAC address
   - Set operation: Install
   - Choose authentication: By Password
   - Enter secure password
4. **Complete Setup** (Step 4)
   - Review configuration
   - Click Finish

### Enrolling a Phone

1. **Add Device** (Devices page â†’ Add Device)
   - Device Name: SEP001122334455
   - Operation: Install
   - Authentication: By Password
   - Password: YourSecurePassword
   - Key Size: 2048
2. **Configure Phone** (on device)
   - Settings â†’ Security â†’ LSC â†’ Update
   - Enter password when prompted
3. **Monitor Progress** (CAPF Audit Logs)
   - Watch for authentication
   - Verify enrollment success
4. **Verify Installation** (Devices page)
   - Device status: Installed
   - Certificate serial number shown
   - Check expiration date

### Adding Server Certificate

1. **Obtain Certificate** (from server)
   ```bash
   openssl s_client -connect server:443 \
     -showcerts | openssl x509 > server.pem
   ```
2. **Upload to TVS** (Certificates â†’ Upload)
   - Select server.pem file
   - Choose roles based on server type
   - Set TTL (default 86400)
3. **Verify Upload** (Certificates list)
   - Certificate appears in list
   - Correct roles assigned
4. **Test Validation** (check logs)
   - Phone queries TVS
   - Validation succeeds

### Backup Procedures

**Weekly Backup:**
1. **TVS Backup** (Admin â†’ TVS Database)
   - Create backup with logs
   - Download backup file
   - Store securely offsite
2. **CAPF Backup** (Admin â†’ CAPF Database)
   - Create backup with certificates
   - Download backup file
   - Secure storage

**Before Major Changes:**
- Always create backup first
- Test restore procedure
- Document changes made

### Troubleshooting with Logs

**Phone Can't Validate Server:**
1. **Check TVS Audit Logs**
   - Filter: Operation = validate
   - Look for certificate hash
   - Check success/failure
2. **If Not Found**:
   - Certificate not uploaded to TVS
   - Upload server certificate
3. **If Failed**:
   - Review error message
   - Check certificate validity
   - Verify roles match

**Phone Enrollment Fails:**
1. **Check CAPF Audit Logs**
   - Filter: Device name
   - Look for authentication attempts
2. **Common Issues**:
   - Authentication failed: Wrong password
   - No pending operation: Not scheduled
   - Timeout: Network connectivity
3. **Resolution**:
   - Update device password
   - Reschedule operation
   - Check network/firewall

## Best Practices

### Security

1. **Use Strong Passwords**
   - Minimum 12 characters
   - Mix of letters, numbers, symbols
   - Unique per device or device group

2. **Regular Backups**
   - Automated daily backups
   - Weekly offsite storage
   - Test restore procedures quarterly

3. **Monitor Audit Logs**
   - Daily review for failures
   - Weekly security audit
   - Alert on unusual patterns

4. **Certificate Lifecycle**
   - Track expiration dates
   - Renew 30 days before expiry
   - Automate renewal where possible

### Operational

1. **Document Everything**
   - Device inventory with MAC addresses
   - Password management (encrypted)
   - Network topology
   - Change log

2. **Test Before Production**
   - Pilot with single device
   - Verify full workflow
   - Document issues and resolutions

3. **Staged Rollout**
   - Enroll devices in batches
   - Monitor each batch
   - Pause if issues arise

4. **Regular Maintenance**
   - Review device status weekly
   - Clean up old audit logs
   - Update documentation
